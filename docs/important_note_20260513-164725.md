# リファクタリング計画: マッチ管理の再設計

作成日: 2026-05-13

---

## 背景・動機

### ドメインモデルの明確化

「マッチ」の定義を以下のように厳密化する。

| 状態 | 扱い |
|---|---|
| replay あり + 動画あり | マッチ ✓ |
| replay あり + 動画なし | マッチ（動画未リンク）✓ |
| 動画のみ（replay なし） | **マッチとして扱わない → 捨てる** |

現在の `pairing.py` は replay のない「孤立動画エントリ」を `has_replay: false` として matches
リストに混入させている。これはドメイン上誤りであり、今回のリファクタリングで修正する。

---

## バックエンド変更

### 1. `match_state.py` — `video_path` フィールド追加

```json
{
  "video_path": null,
  "trimmed_video_path": null,
  "trim_start_offset_sec": null,
  "kill_offsets_in_trimmed": [],
  "has_summary": false,
  "kill_compilation_path": null,
  "kill_times_in_match": [],
  "match_result": null
}
```

`video_path` はサイドカーへの**単純な上書き**。自動リンク・手動リンクともにこのフィールドに書き込む。
リンク解除（null にする）も可能だが、元のパスを知っていれば再入力で戻せるため、特別な Undo 機能は不要。

### 2. `pairing.py` — 孤立動画エントリの生成を削除

`pair()` の末尾にある「orphan videos」セクション（replay なしエントリの生成部分）を削除する。
`has_replay` フラグは常に `true` になるため、後続処理から参照箇所も削除できる。

### 3. `main.py` — `_merge_state()` の video 解決ロジック変更

サイドカーの `video_path` が非 null であればそれを優先し、null であれば `pair()` の自動マッチ結果を使う。
どちらも null なら「動画なし」。

```python
# 優先順位: サイドカーオーバーライド > 自動ペアリング結果
sidecar_video_path = state.get("video_path")
if sidecar_video_path:
    # video_path からファイル情報を組み立て
else:
    # pair() が返した video をそのまま使う
```

### 4. 新エンドポイント: `PUT /api/suite/matches/{id}/video`

```
Body: { "videoPath": "H:\\...\\Replay_xxx.mp4" | null }
```

- 指定パスをサイドカーの `video_path` に保存
- null を渡すとリンク解除
- 保存後 `_matches_cache` を無効化

### 5. 新エンドポイント: `POST /api/suite/matches/{id}/auto-link-video`

- 録画フォルダをスキャンし、既存のペアリングロジック（時刻ウィンドウ一致）で最良の動画を検索
- 見つかった場合: `video_path` に保存し、candidates API を呼び出して `trim_start_offset_sec` + `kill_offsets_in_trimmed` もサイドカーへ保存
- 見つからない場合: `404` を返す（フロントエンドは手動リンク UI を表示）

### 6. 新エンドポイント: `GET /api/suite/videos`

録画フォルダ内の動画ファイル一覧を返す（手動リンク用ドロップダウンのデータソース）。

```json
[
  { "path": "H:\\...\\Replay_2026-05-12_22-23-39.mp4", "filename": "...", "durationSec": 1554.3, "mtime": "..." }
]
```

### 7. `post_match_automation` の拡張

現在の自動フロー終了後に動画リンクとオフセット計算を追加する。

```
post_match_automation 呼び出し
  ├─ リプレイ解析 → kill_times_in_match + match_result + has_summary
  ├─ 動画を自動リンク（auto-link-video ロジックを流用）
  │    ├─ 成功 → video_path + trim_start_offset_sec + kill_offsets_in_trimmed をサイドカーへ
  │    └─ 失敗 → video_path = null のまま（以降は手動）
  └─ レスポンスに videoLinked: true/false を含める
```

ffmpeg による trimmed ファイル生成・Kill 集生成は重い処理のため、引き続き手動トリガーのまま。

### 8. バグ修正: kill 抽出フィールドパス

`summarize_match` / `post_match_automation` 両方で誤ったフィールドパスを使用している。

```python
# 誤り（現状）: 常に None → kill 0 件
elim.get("Eliminator")

# 正しい（candidates.py と同じ）
((elim.get("EliminatorInfo") or {}).get("Id") or "")
```

### 9. 重複コードのヘルパー抽出

`summarize_match` と `post_match_automation` が共有する約 40 行のリプレイ解析 + kill 抽出処理を
ヘルパー関数 `_extract_kills_from_replay(replay_path, user_player_id)` に切り出す。

## フロントエンド変更

### 10. `suiteCore.ts` — 型・API クライアントの更新

- `Match.hasReplay` を削除（常に `true` のため不要）
- `Match` に `video` フィールドの取得元変更（サイドカー `video_path` 優先はバックエンド側で処理済み、型は変わらず）
- 追加 API クライアントメソッド:
  - `linkVideo(id, path | null)`
  - `autoLinkVideo(id)`
  - `listVideos()`

### 11. `MatchLibrary.tsx` — `hasReplay: false` ガード削除

`MatchCard` から `hasReplay` チェックを削除。カード表示の整理。

### 12. `MatchDetail.tsx` — 「録画動画」セクションの改修

**動画ありの場合（既存）:**
- サムネイル表示、パス表示
- 「パスを変更」ボタン → 手動リンク UI へ

**動画なしの場合（新規）:**
1. 「自動検出」ボタン → `autoLinkVideo()` を呼び出し
   - 成功: 動画情報表示 + `kill_offsets_in_trimmed` が計算済みであればすぐ Kill 集生成へ進める
   - 失敗: 「見つかりませんでした」メッセージ → 手動リンクへ誘導
2. 「手動リンク」 → 録画フォルダの動画一覧ドロップダウン（`listVideos()` のデータ）
   - 選択後 `linkVideo()` を呼び出し
   - リンク後、`kill_offsets_in_trimmed` が未計算であれば「キルオフセットを計算」ボタンを表示

## OBS タイミング問題時の手動フォールバックフロー

```
試合終了 → post_match_automation 自動実行
  ↓
 kill_times_in_match + match_result + has_summary = 保存済み
 video_path = null（OBS がまだ保存完了していない等）
  ↓
ユーザーが後でマッチ詳細ページを開く
  ↓
「自動検出」を試みる → 成功すれば完了
  ↓
失敗した場合:「手動リンク」ドロップダウンで動画を選択
  ↓
「キルオフセットを再計算」ボタン → compute_kills → kill_offsets_in_trimmed をサイドカーへ
  ↓
通常の trimmed 生成 → Kill 集生成フローへ
```
